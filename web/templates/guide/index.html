Welcome to the GNIZA Getting Started Guide

This step-by-step tutorial will walk you through setting up GNIZA from scratch — from installation to your first automated backup. Follow the sections in order, or jump to any topic using the table of contents.

0 Introduction — What is GNIZA?

GNIZA is a complete Linux backup solution that can run as a stand-alone backup tool or as a centralized backup server managing multiple machines. It is built on top of rsync with --link-dest for incremental hardlink deduplication, meaning each snapshot looks like a full backup but only changed files consume extra disk space.

GNIZA provides three interfaces to manage your backups:

  • Web Dashboard — a full-featured browser UI (what you are using now)
  • TUI — a terminal-based interface built with Python Textual
  • CLI — command-line flags for scripting and automation

Key features at a glance:

  • Agentless — no software needed on source machines (SSH pull)
  • Incremental snapshots with hardlink deduplication
  • Atomic snapshots (.partial rename on success)
  • Built-in MySQL/MariaDB dump support
  • Multiple source types: local, SSH, S3, Google Drive
  • Multiple destination types: local, SSH, S3, Google Drive
  • Automatic retention and pruning
  • Email notifications
  • Concurrency control via flock

1 Why GNIZA?

If you have ever set up backups with raw rsync, tar, or tools like Dirvish, you know how much manual plumbing is involved — writing wrapper scripts, managing retention, setting up cron jobs, handling MySQL dumps separately, and hoping nothing silently breaks.

GNIZA replaces all of that with a single integrated system:

Manual Approach GNIZA
Write rsync wrapper scriptsDeclarative INI config files
DIY hardlink rotationAutomatic --link-dest dedup
Separate mysqldump cronBuilt-in MySQL dump before file sync
Manual crontab entriesSchedule management via UI
Custom log parsingStructured job logs with status
No visibility into progressReal-time Running Tasks view
Manual cleanup of old backupsAutomatic retention with pruning
Hope it worksEmail alerts on success/failure

The result: each backup snapshot appears as a complete directory tree (easy to browse and restore from), but only files that actually changed since the last run consume additional disk space.

2 Installation

GNIZA can be installed with a single command. The installer detects whether you are running as root or a regular user and adjusts paths accordingly.

As root (system-wide)

curl -sL https://git.linux-hosting.co.il/shukivaknin/gniza4linux/raw/branch/main/scripts/install.sh | sudo bash

This installs to:

  • Application: /usr/local/gniza
  • Configuration: /etc/gniza/
  • Logs: /var/log/gniza/

As regular user

curl -sL https://git.linux-hosting.co.il/shukivaknin/gniza4linux/raw/branch/main/scripts/install.sh | bash

This installs to:

  • Application: ~/.local/share/gniza
  • Configuration: ~/.config/gniza/
  • Logs: ~/.local/state/gniza/log/

Prerequisites

The installer handles most dependencies automatically. You will need:

  • rsync — the core transfer engine
  • ssh — for remote sources and destinations
  • python3 + pip — for the web dashboard and TUI
  • curl or wget — for the installer itself
Tip: After installation, access the web dashboard at http://your-server:2323. The first time you log in, use the API key shown during installation (or set WEB_API_KEY in gniza.conf).

3 Setting up SSH Keys

For backing up remote machines (or storing backups on a remote server), you need passwordless SSH access. This is the standard approach for automated backups.

Generate a key pair

ssh-keygen -t ed25519 -C "gniza-backup"

Press Enter to accept the default path. Do not set a passphrase (automated backups cannot type passwords).

Copy the public key to the remote machine

ssh-copy-id user@remote-server

Replace user@remote-server with the actual username and hostname of the machine you want to back up (or store backups on).

Test the connection

ssh user@remote-server

You should be logged in without being prompted for a password. If it works, you are ready to configure GNIZA sources and destinations.

Tip: GNIZA also supports password-based SSH authentication via sshpass if key-based auth is not an option. Set SOURCE_AUTH_METHOD=password and provide the password in the source configuration. However, SSH keys are strongly recommended for security and reliability.

4 Configuring a Source (What to Back Up)

A source (also called a target) defines what data GNIZA should back up. You can configure sources through any of the three interfaces:

  • Web: Navigate to Sources > Add
  • TUI: Sources > Add
  • CLI: Create a .conf file in the targets directory

Example: Local source

Back up a directory on the same machine where GNIZA is installed:

targets.d/home.conf

[target]
NAME=home
SOURCE_TYPE=local
FOLDERS=/home/user
ENABLED=yes

Example: Remote SSH source

Pull backups from a remote server over SSH (agentless — no GNIZA needed on the remote):

targets.d/webserver.conf

[target]
NAME=webserver
SOURCE_TYPE=ssh
SOURCE_HOST=192.168.1.100
SOURCE_USER=backup
SOURCE_PORT=22
SOURCE_AUTH_METHOD=key
FOLDERS=/var/www,/etc
ENABLED=yes

Key fields:

  • NAME — a unique identifier for this source
  • SOURCE_TYPE — local, ssh, s3, or gdrive
  • FOLDERS — comma-separated list of paths to back up
  • ENABLED — yes or no to toggle the source

5 Configuring a Destination (Where to Store Backups)

A destination (also called a remote) defines where GNIZA stores the backup snapshots. Configure via:

  • Web: Navigate to Destinations > Add
  • TUI: Destinations > Add
  • CLI: Create a .conf file in the remotes directory

Example: SSH destination

Store backups on a remote backup server:

remotes.d/backup-server.conf

[remote]
NAME=backup-server
TYPE=ssh
HOST=backup.example.com
USER=backup
PORT=22
AUTH_METHOD=key
BASE=/home/backup/gniza

Example: Local destination

Store backups on a local drive or mounted USB disk:

remotes.d/usb-drive.conf

[remote]
NAME=usb-drive
TYPE=local
BASE=/mnt/backup-drive/gniza

Key fields:

  • NAME — a unique identifier for this destination
  • TYPE — local, ssh, s3, or gdrive
  • BASE — the root directory where snapshots will be stored
Tip: S3 and Google Drive destinations are also supported. Configure them through the web dashboard for the easiest setup experience.

6 Running Your First Backup

With at least one source and one destination configured, you are ready to run your first backup.

Via the web dashboard

  1. Navigate to Backup
  2. Select the source you want to back up
  3. Click Run Backup
  4. Monitor progress on the Running Tasks page

Via the TUI

  1. Launch the TUI: gniza --tui
  2. Go to the Backup screen
  3. Select your source and run

Via the CLI

gniza --cli --backup home

Replace home with the NAME of the source you want to back up.

What happens during a backup?

  1. GNIZA creates a .partial snapshot directory on the destination
  2. rsync transfers files using --link-dest to hardlink unchanged files from the previous snapshot
  3. If MySQL backup is enabled, databases are dumped before file transfer
  4. On success, the .partial directory is renamed to the final timestamped name (atomic)
  5. A job log entry is recorded with the result
Note: The first backup will transfer all files (there is no previous snapshot to hardlink against). This may take a while depending on the data size and network speed. Subsequent backups will be much faster since only changed files are transferred.

7 MySQL / MariaDB Backup

GNIZA can automatically dump MySQL or MariaDB databases as part of the backup process. The dumps run before the file transfer, ensuring database consistency.

Enabling MySQL backup

In your source configuration, enable MySQL and provide the connection details:

[target]
NAME=webserver
SOURCE_TYPE=ssh
SOURCE_HOST=192.168.1.100
SOURCE_USER=backup
FOLDERS=/var/www,/etc
MYSQL_ENABLED=yes
MYSQL_USER=backup
MYSQL_PASS=secretpassword
MYSQL_DATABASES=all
MYSQL_GRANTS=yes
ENABLED=yes

Options:

  • MYSQL_ENABLED=yes — turn on database dumping
  • MYSQL_USER / MYSQL_PASS — MySQL credentials
  • MYSQL_DATABASES=all — dump all databases, or specify a comma-separated list
  • MYSQL_GRANTS=yes — also back up user grants

For SSH sources, mysqldump runs on the remote machine. The dump files are stored alongside the file snapshots in the backup destination.

Tip: Create a dedicated MySQL user for backups with read-only privileges and LOCK TABLES permission for consistent dumps.

8 Scheduling Automatic Backups

Schedules define when backups run automatically. Configure them via:

  • Web: Navigate to Schedules > Add
  • TUI: Schedules > Add
  • CLI: Create a .conf file in the schedules directory

Example schedule

schedules.d/nightly.conf

[schedule]
NAME=nightly
SCHEDULE=daily
TIME=02:00
TARGETS=home,webserver
ACTIVE=yes

Key fields:

  • NAME — unique schedule identifier
  • SCHEDULE — frequency: daily, weekly, or monthly
  • TIME — when to run (24-hour format)
  • TARGETS — comma-separated list of source names to back up
  • ACTIVE — yes or no to enable/disable

Schedules are managed by the GNIZA daemon. Each schedule can also have its own RETENTION_COUNT to override the global default (see next section).

9 Retention and Cleanup

GNIZA automatically prunes old snapshots based on retention settings. This prevents your backup storage from growing indefinitely.

How retention works

  • After each successful backup, GNIZA checks how many snapshots exist for that source
  • If the count exceeds the retention limit, the oldest snapshots are deleted
  • Thanks to hardlink dedup, deleting old snapshots only frees space for files that have changed since

Configuration

Retention is configured at two levels:

  • Global default: Set RETENTION_COUNT in gniza.conf (e.g., RETENTION_COUNT=7)
  • Per-schedule override: Add RETENTION_COUNT to any schedule config to override the global value for that schedule

Example: Weekly schedule with 4-week retention

[schedule]
NAME=weekly
SCHEDULE=weekly
TIME=03:00
TARGETS=home,webserver
RETENTION_COUNT=4
ACTIVE=yes

Snapshot pinning

You can pin specific snapshots to protect them from automatic pruning. Pinned snapshots are never deleted by the retention system — useful for milestone backups (before a major upgrade, for example).

10 Browsing and Restoring from Backups

GNIZA makes it easy to browse and restore from your backup snapshots.

Via the web dashboard

  1. Navigate to Snapshots
  2. Select the source and snapshot date
  3. Use the file browser to navigate the backup contents
  4. Download individual files or folders
  5. For full restore: go to Restore, select the snapshot, and provide the target path

Via the CLI

gniza --cli --restore home --snapshot 2026-03-10_020000 --target /home/user

Manual restore

Since GNIZA snapshots are regular directories with regular files (and hardlinks), you can always restore manually:

# Copy a single file
cp /path/to/backup/home/2026-03-10_020000/home/user/document.txt ~/document.txt

# Restore a full directory
rsync -avP /path/to/backup/home/2026-03-10_020000/var/www/ /var/www/
Tip: Test your restore process periodically. A backup you cannot restore from is not a backup.

11 Monitoring

GNIZA provides several tools to keep track of your backup health:

Logs

The Logs page shows all job logs with their status (success, failure, running). Each log entry includes timestamps, file counts, and transfer statistics.

Email notifications

Configure email settings in Settings to receive notifications on backup success or failure. The Email Log page shows all sent notifications.

Running Tasks

The Running Tasks page shows real-time progress of active backup and restore jobs. A badge in the sidebar indicates how many jobs are currently running.

Health checks

The Health section in Settings monitors the overall state of your backup system — checking for stale sources, failed schedules, and storage capacity.

12 Tips and Best Practices

Exclude unnecessary files

Save space and time by excluding directories like .cache, node_modules, __pycache__, .tmp, and /proc. Configure excludes in the source settings.

Use SSH keys

Always prefer SSH key authentication over passwords for automated backups. It is more secure and eliminates the risk of password expiry breaking your backup schedule.

Test restores regularly

Schedule a periodic test restore to verify your backups are actually usable. Use the Restore page to restore to a temporary location and spot-check files.

Monitor disk space

Keep an eye on available space on your backup destinations. While hardlink dedup is efficient, long retention periods with high churn can still fill disks.

Set sensible retention

Balance between having enough history and not wasting storage. A common setup: daily backups with 7-day retention, weekly with 4-week retention.

Pin important snapshots

Before major system upgrades or deployments, pin the current snapshot so retention does not automatically delete it.
